# Protobuf Runtime Strategy Across the Pinba Stack

The engine (server) and the PHP extension (client) both speak Protocol Buffers,
but they deliberately use **different protobuf runtimes**. The guiding principle
is: **unify the contract, not the runtime.**

## The shared contract: `pinba.proto`

Both repositories ship a copy of `pinba.proto` (proto2, `package Pinba`,
message `Request`). It is the single stable contract between client and server:

- Fields 1–23 have been stable since 1.1.0 (2015).
- Rule: only **append** fields; never renumber or retype an existing field.
- Wire compatibility depends only on field numbers and types, so the two copies
  must stay field-identical even though their generated code differs.

The on-wire format is determined entirely by this schema, which is why the two
sides can use unrelated runtimes and still interoperate.

## Engine: C++ protobuf + a compatibility shim

- The engine is a MySQL 8 plugin built with CMake and C++23.
- It decodes packets with the **C++ Protocol Buffers runtime** (`libprotobuf`,
  found via `find_package(Protobuf)`, `>= 3.0`). CMake generates `pinba.pb.{h,cc}`
  from `pinba.proto` via `protobuf_generate_cpp`.
- `src/pinba.pb-c.cc` is a **shim**: it converts a decoded C++ `Pinba::Request`
  into a flat C-style `Pinba__Request` struct declared in the hand-written
  `src/pinba.pb-c.h`, so the legacy data path (`data.cc`, `ha_pinba.cc`,
  `pool.cc`) did not have to be rewritten during the MySQL 8 migration.
- **Rationale:** MySQL itself ships and links the C++ protobuf runtime, so reusing
  it keeps the toolchain and ABI aligned with the host server.
- This is **not** the protobuf-c (C) runtime — only the struct shape resembles it.
  The old vendored `src/protobuf-c.h` (protobuf-c 2.0.0) was left over from before
  the migration, was included by nothing, and was removed as dead weight.

## Extension: protobuf-c (pure C)

- The PHP extension is a lean C extension built with `phpize`/autoconf.
- It uses the **protobuf-c (C) runtime**, linked from the system library
  `libprotobuf-c` (`>= 1.0`) via `pkg-config` in `config.m4`. Bindings
  (`pinba-pb-c.c` / `pinba.pb-c.h`) are generated by `protoc-c` (protoc-gen-c).
- **Rationale:** a PHP C extension should not drag in the heavy C++ protobuf
  runtime; protobuf-c is small and idiomatic for C.
- The extension previously **vendored** a 2008-era copy of the protobuf-c runtime.
  It now depends on the distribution package instead, matching Debian policy
  against embedded code copies and gaining distro security/maintenance updates.

## Fields 22/23: per-timer resource usage

`timer_ru_utime` (22) and `timer_ru_stime` (23) carry per-timer CPU usage
(user-mode and system-mode time), as `repeated float` arrays parallel to
`timer_value`/`timer_hit_count` (element *i* belongs to timer *i*).

- Introduced in the engine on 2013-11-12 by Antony Dovgal,
  commit "add rusage to timers and timer reports". The same change added engine
  storage and exposed `ru_utime`/`ru_stime` columns in timer/tag report tables.
- The extension has always collected and sent them (via `getrusage()` per timer,
  and via the explicit `rusage` argument of `PinbaClient::setTimer`/`addTimer`).

## Drift hazard and the drift guard

The extension's checked-in `pinba.proto` once lagged its own generated bindings:
it was missing fields 22/23 while the generated code and the engine already had
them. Regenerating from the stale schema would have silently dropped per-timer
rusage from the wire and broken the contract.

Mitigations now in place:

- The extension's `pinba.proto` was resynced with the generated code and the
  engine.
- The extension CI has a **drift guard** that regenerates the bindings with
  `protoc-c` and fails if the committed files differ from `pinba.proto`.
- Wire neutrality of the runtime migration was verified by comparing the
  serialized packet for a deterministic input before and after: byte-for-byte
  identical (same SHA-256).

Keep the two `pinba.proto` copies field-compatible; if a field is added on one
side, mirror it on the other and regenerate both runtimes' bindings.
